# Codex 不具合解析

## 対象の判定

`$ARGUMENTS` の内容に応じて解析対象を決定する:

- **Issue 番号** (例: `#34`, `34`): `gh issue view` で症状・再現手順・観察結果を取得
- **テキスト**: 症状の説明としてそのまま解析対象とする
- **ファイルパス**: 問題が発生しているコードを起点に解析

## 解析実行手順

1. 不具合の情報を収集する
   - Issue の場合: 症状、再現手順、観察結果、コメントの経緯
   - 関連するソースコード、ログ、テスト結果
   - 直近の変更 (`git log`, `git diff`) で原因となりうるコミット
2. 問題の層を特定する (後述の「層の切り分け」参照)
3. pybricks に同等機能がある場合、同種の問題がどう対処されているか確認する
4. 収集した情報と以下の「解析観点」を Codex プロンプトに構成する
5. `mcp__codex__codex` ツールで Codex に解析を依頼する
6. Codex の結果を整理してユーザーに報告する

## 解析観点

### 1. 根本原因の仮説列挙

Codex に以下を求める:

- **複数の仮説を確度順に列挙** — 1つに絞らず、考えられる原因を網羅する
- **各仮説について根拠となるコード箇所を特定** — ファイルパスと行番号
- **各仮説の検証方法を提案** — 最小限の実験やログ追加で切り分けられる手順

### 2. 層の切り分け

このプロジェクトは複数層にまたがるため、問題がどの層に属するか明確にする:

| 層 | 例 |
|---|---|
| HW / 電気的要因 | USB CDC タイミング、DMA 転送、信号品質、電源ノイズ |
| MCU ペリフェラル | レジスタ設定ミス、割込み競合、クロック設定 |
| NuttX カーネル | ドライババグ、スケジューラ挙動、クリティカルセクション |
| ボードドライバ | `boards/spike-prime-hub/src/` のコード |
| アプリケーション | `apps/` のコード |
| テスト基盤 | `tests/conftest.py`、pexpect/pyserial の挙動 |

### 3. HW 起因の可能性を必ず検討する

組み込み特有の問題は SW だけでは説明できないことが多い。以下を Codex に必ず検討させる:

- タイミング依存の問題か (race condition、バッファフラッシュ遅延)
- 割込み優先度やクリティカルセクションに関連するか
- DMA 転送完了と CPU アクセスの競合はないか
- USB CDC/ACM の仕様上の制約に起因しないか
- 電源状態 (USB/バッテリー) で挙動が変わるか

### 4. pybricks での対処確認

pybricks に同等機能がある場合:

- pybricks で同種の問題が報告・修正されていないか
- pybricks の実装で同じ問題をどう回避しているか
- pybricks と本プロジェクトの実装差分が原因になっていないか

pybricks ソースの参照先: `pybricks/` サブモジュール

### 5. 再現性の分析

- 確定的に再現するか、間欠的 (flaky) か
- 再現条件を狭められるか (特定のテスト順序、タイミング、HW 状態)
- 単独実行 vs バッチ実行で挙動が変わるか

## Codex へのプロンプト構成

Codex に送るプロンプトには以下を含める:

1. 症状の正確な記述 (エラーメッセージ、ログ、期待値 vs 実際値)
2. 再現手順と再現率
3. 関連するソースコード (問題箇所の周辺)
4. プロジェクトのコンテキスト (MCU: STM32F413, RTOS: NuttX 12.13.0, テスト基盤: pytest + pexpect + pyserial)
5. 上記の解析観点のうち該当するものを明示的に指示
6. 「仮説は確度順に並べ、各仮説に検証方法を付けること」という指示
