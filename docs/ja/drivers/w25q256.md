# W25Q256 SPI NOR Flash + LittleFS

SPIKE Prime Hub の外付け Winbond **W25Q256 (32 MB)** にアクセスする board-local MTD ドライバ。NuttX submodule への変更を最小化するため、上流 `drivers/mtd/w25.c` (3-byte addressing 限定) は触らず、専用ドライバを `boards/spike-prime-hub/src/stm32_w25q256.c` に新規実装。LittleFS で `/mnt/flash` にマウントし、ファイル永続化を提供する。

## ハードウェア構成

| 信号 | ピン / 周辺機能 | 説明 |
|------|----------------|------|
| SCK | PB13 (AF5) | SPI2_SCK |
| MISO | PC2 (AF5) | SPI2_MISO |
| MOSI | PC3 (AF5) | SPI2_MOSI |
| /CS | PB12 (GPIO) | ソフトウェア NSS、idle HIGH |
| DMA RX | DMA1 Stream3 Channel 0 | NuttX SPI ドライバが自動取得 |
| DMA TX | DMA1 Stream4 Channel 0 | NuttX SPI ドライバが自動取得 |
| SPI mode | Mode 0 (CPOL=0, CPHA=0) | 8-bit, MSB first |
| クロック | 24 MHz (APB1 48 MHz / 2) | W25Q256 max 50 MHz の半分 |

ピン配線は pybricks `lib/pbio/platform/prime_hub/platform.c:867-942` と完全一致。SPI2_MISO/MOSI は STM32F413 のもう 1 つの選択肢である PB14/PB15 ではなく PC2/PC3。

## 4-byte 専用コマンド

W25Q256 は 32 MB のため 4-byte アドレッシングが必要。本ドライバは address mode register に依存しないよう、常に dedicated 4-byte opcode を使用する (pybricks 同等):

| 操作 | コマンド | 用途 |
|---|---|---|
| Read JEDEC ID | `0x9F` | 起動時のチップ検証 (`0xEF4019`) |
| Write Enable | `0x06` | 各 program/erase の前 |
| Read Status Register-1 | `0x05` | BUSY ビット監視 |
| Fast Read 4-byte | `0x0C` | `[cmd][a3][a2][a1][a0][dummy][data...]` |
| Page Program 4-byte | `0x12` | 256 B/page |
| Sector Erase 4-byte | `0x21` | 4 KB |
| Block Erase 4-byte | `0xDC` | 64 KB (将来の保守ユーティリティ用) |
| Chip Erase | `0xC7` | `MTDIOC_BULKERASE` ハンドラから呼ぶ |

## MTD インターフェース

`struct mtd_dev_s` の以下メソッドを実装:

| メソッド | 内容 |
|---|---|
| `erase` | 4KB sector 単位の消去 (`Sector Erase 0x21`) |
| `bread` | block 単位読出し (Fast Read 連続) |
| `bwrite` | block 単位書込み (Page Program ループ) |
| `read` | byte 単位読出し (LittleFS は使わないが互換のため実装) |
| `ioctl` | 下記 ioctl を処理 |

サポート ioctl:

| ioctl | 用途 |
|---|---|
| `MTDIOC_GEOMETRY` | `blocksize=256` (page), `erasesize=4096` (sector), `neraseblocks=8192` (chip 全体 32 MB / 4 KB) を返す。LittleFS の `prog_size` / `block_size` を決定 |
| `BIOC_PARTINFO` | partition 情報を返す (`mtd_partition()` および LittleFS bind で使用) |
| `MTDIOC_ERASESECTORS` | `mtd_erase_s` で指定された範囲を消去 |
| `MTDIOC_BULKERASE` | Chip Erase (`0xC7`) で全消去 — 保守ユーティリティ向け |
| `MTDIOC_ERASESTATE` | 消去後値 `0xFF` を返す |

## パーティションレイアウトと公開デバイス

```
0x000000 ┌───────────────────────────────────┐
         │  LEGO bootloader area (512 KB)    │  予約
0x080000 ├───────────────────────────────────┤
         │  pybricks block device (256 KB)   │  予約
0x0FF000 ├───────────────────────────────────┤
         │  Update key (4 KB)                │  予約
0x100000 ├───────────────────────────────────┤
         │                                   │
         │  /dev/mtdblock0 → /mnt/flash      │  31 MB (LittleFS)
         │                                   │
0x1FFFFFF└───────────────────────────────────┘
```

- 先頭 1 MB (`0x000000-0x0FFFFF`) は予約 (LEGO ブートローダー / pybricks data 領域を保護)
- `0x100000` 以降の 31 MB を `mtd_partition()` で切り出し、`/dev/mtdblock0` として登録
- **full chip raw (`/dev/mtd0`) は公開しない** — user 空間から LEGO 領域を消去できないようにするため。デバッグで raw access が必要な場合は `stm32_w25q256.c` 内の `register_mtddriver("/dev/mtd0", mtd_full, ...)` のコメントアウトを外して再ビルド (defconfig 永続化はしない)

## 起動フロー

`board_late_initialize()` から呼ばれる `stm32_bringup()` 内で `#ifdef CONFIG_STM32_SPI2` ブロックで `stm32_w25q256_initialize()` を呼ぶ。

1. CS GPIO (PB12, idle HIGH) を初期化
2. `stm32_spibus_initialize(2)` で SPI2 を取得
3. `w25q256_initialize(spi)` で MTD オブジェクトを作成 — JEDEC ID 検証 (`0xEF4019` でなければ `-ENODEV` 返却)
4. `mtd_partition()` で FS パーティション (`0x100000~`, 31 MB) を切り出し
5. `register_mtddriver("/dev/mtdblock0", ...)` で公開
6. `nx_mount("/dev/mtdblock0", "/mnt/flash", "littlefs", 0, NULL)` で mount。`-EFAULT` (corrupt) または `-EINVAL` (新品の未フォーマットチップ) を検出したら `"forceformat"` で format + remount

!!! note "autoformat ではなく forceformat フォールバック"
    NuttX LittleFS の `"autoformat"` オプションは `lfs_mount` が `-EFAULT` を返したときのみ format に進む。だが新品チップ (全 0xff) の場合 LittleFS は superblock が見つからず `-EINVAL` (= `LFS_ERR_INVAL`) を返すため autoformat は発火しない (`nuttx/fs/littlefs/littlefs/lfs.c:4259`)。本ドライバは初回 mount を `NULL` data で試行 → `-EFAULT` または `-EINVAL` の場合のみ `"forceformat"` でリトライする。SPI 通信エラー (`-EIO`) では format に進まない (誤消去防止)。

### 失敗時の挙動 (non-fatal)

`stm32_w25q256_initialize()` が失敗 (JEDEC mismatch、SPI エラー、mount 失敗 etc.) した場合は `syslog(LOG_ERR, ...)` でエラー報告のみ行い、`stm32_bringup()` 全体は継続する。`/mnt/flash` は不在のまま NSH が起動する。失敗を観測するには `dmesg` で `W25Q256:` の行を確認する。

## NVIC IRQ 優先度

`stm32_bringup.c` の "Issue #36 epsilon plan" ブロックで `STM32_IRQ_SPI2` / `STM32_IRQ_DMA1S3` / `STM32_IRQ_DMA1S4` を **NVIC 0xB0** に設定。USB OTG FS と同居。pybricks の相対順序 (Sound 4 > Flash 5/6 > USB 6 > ADC/TLC 7) を 0x80-0xF0 帯に圧縮した ε スキームに従う。詳細は [DMA/IRQ 割当](../hardware/dma-irq.md) を参照。

## CONFIG 関連

usbnsh defconfig:

```
# SPI2 + DMA
CONFIG_STM32_SPI2=y
CONFIG_STM32_SPI2_DMA=y

# MTD
CONFIG_MTD=y
CONFIG_MTD_BYTE_WRITE=y
CONFIG_MTD_PARTITION=y

# LittleFS
CONFIG_FS_LITTLEFS=y

# md5 (テスト/検証用)
CONFIG_NETUTILS_CODECS=y
CONFIG_CODECS_HASH_MD5=y
```

## 既知の制約

- **DMA priority は MEDIUM 据え置き** — pybricks の HIGH 設定とは異なる。詳細は [DMA/IRQ 割当](../hardware/dma-irq.md) の Note を参照
- **SPI bus は SPI2 単独使用** — 他に SPI2 デバイスを追加する場合は `stm32_spi2select` の devid 分岐を拡張
- **書込み速度の制約** — LittleFS は内部に metadata 更新のため複数 erase + write を伴う。ファイル新規作成時に数秒かかることがある (1 MB 全体での実測例: ~1MB/45 秒 ≈ 23 KB/s, Zmodem オーバーヘッド込み)
