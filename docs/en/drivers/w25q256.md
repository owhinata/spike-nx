# W25Q256 SPI NOR Flash + LittleFS

Board-local MTD driver for the SPIKE Prime Hub's external Winbond **W25Q256 (32 MB)** NOR flash.  To minimize NuttX submodule changes, the upstream `drivers/mtd/w25.c` (3-byte addressing only) is left untouched and a dedicated driver is implemented in `boards/spike-prime-hub/src/stm32_w25q256.c`.  The chip is mounted as LittleFS at `/mnt/flash` and provides persistent file storage.

## Hardware Configuration

| Signal | Pin / Peripheral | Description |
|------|----------------|------|
| SCK | PB13 (AF5) | SPI2_SCK |
| MISO | PC2 (AF5) | SPI2_MISO |
| MOSI | PC3 (AF5) | SPI2_MOSI |
| /CS | PB12 (GPIO) | Software NSS, idle HIGH |
| DMA RX | DMA1 Stream3 Channel 0 | Auto-acquired by NuttX SPI driver |
| DMA TX | DMA1 Stream4 Channel 0 | Auto-acquired by NuttX SPI driver |
| SPI mode | Mode 0 (CPOL=0, CPHA=0) | 8-bit, MSB first |
| Clock | 24 MHz (APB1 48 MHz / 2) | Half of W25Q256's 50 MHz max |

The pin wiring matches pybricks `lib/pbio/platform/prime_hub/platform.c:867-942` exactly.  SPI2_MISO/MOSI are on **PC2/PC3**, not the more common PB14/PB15 alternative for STM32F413.

## 4-Byte Dedicated Commands

The W25Q256 is 32 MB and requires 4-byte addressing.  This driver always uses dedicated 4-byte opcodes so it does not depend on the device's address-mode register state (matching pybricks):

| Operation | Opcode | Use |
|---|---|---|
| Read JEDEC ID | `0x9F` | Chip identification at boot (`0xEF4019`) |
| Write Enable | `0x06` | Before each program/erase |
| Read Status Register-1 | `0x05` | Poll the BUSY bit |
| Fast Read 4-byte | `0x0C` | `[cmd][a3][a2][a1][a0][dummy][data...]` |
| Page Program 4-byte | `0x12` | 256 B per page |
| Sector Erase 4-byte | `0x21` | 4 KB |
| Block Erase 4-byte | `0xDC` | 64 KB (reserved for future maintenance utilities) |
| Chip Erase | `0xC7` | Invoked from the `MTDIOC_BULKERASE` handler |

## MTD Interface

`struct mtd_dev_s` methods implemented:

| Method | Purpose |
|---|---|
| `erase` | Erase a range of 4 KB sectors (`Sector Erase 0x21`) |
| `bread` | Block read (continuous Fast Read) |
| `bwrite` | Block write (page-program loop) |
| `read` | Byte-level read (unused by LittleFS, kept for compatibility) |
| `ioctl` | Handles the ioctls below |

Supported ioctls:

| ioctl | Purpose |
|---|---|
| `MTDIOC_GEOMETRY` | Returns `blocksize=256` (page), `erasesize=4096` (sector), `neraseblocks=8192` (whole chip = 32 MB / 4 KB).  Drives LittleFS's `prog_size` / `block_size` |
| `BIOC_PARTINFO` | Returns partition info (used by `mtd_partition()` and the LittleFS bind path) |
| `MTDIOC_ERASESECTORS` | Erase a range described by `mtd_erase_s` |
| `MTDIOC_BULKERASE` | Whole-chip erase via `0xC7` — for maintenance utilities |
| `MTDIOC_ERASESTATE` | Returns `0xFF` (post-erase value) |

## Partition Layout and Exposed Devices

```
0x000000 ┌───────────────────────────────────┐
         │  LEGO bootloader area (512 KB)    │  Reserved
0x080000 ├───────────────────────────────────┤
         │  pybricks block device (256 KB)   │  Reserved
0x0FF000 ├───────────────────────────────────┤
         │  Update key (4 KB)                │  Reserved
0x100000 ├───────────────────────────────────┤
         │                                   │
         │  /dev/mtdblock0 → /mnt/flash      │  31 MB (LittleFS)
         │                                   │
0x1FFFFFF└───────────────────────────────────┘
```

- The first 1 MB (`0x000000-0x0FFFFF`) is reserved (protects the LEGO bootloader and pybricks data area).
- The 31 MB region from `0x100000` is split off via `mtd_partition()` and registered as `/dev/mtdblock0`.
- **The full-chip raw device (`/dev/mtd0`) is not exposed** so user space cannot accidentally erase the LEGO area.  For deep debugging, uncomment the `register_mtddriver("/dev/mtd0", mtd_full, ...)` line in `stm32_w25q256.c` and rebuild — but do not commit that change to defconfig.

## Bring-up Flow

`stm32_w25q256_initialize()` is invoked from `stm32_bringup()` (called by `board_late_initialize()`) inside a `#ifdef CONFIG_STM32_SPI2` block.

1. Configure CS GPIO (PB12, idle HIGH).
2. `stm32_spibus_initialize(2)` to obtain the SPI2 bus.
3. `w25q256_initialize(spi)` creates the MTD object and verifies the JEDEC ID (returns `-ENODEV` if not `0xEF4019`).
4. `mtd_partition()` carves out the FS region (`0x100000~`, 31 MB).
5. `register_mtddriver("/dev/mtdblock0", ...)` exposes it.
6. `nx_mount("/dev/mtdblock0", "/mnt/flash", "littlefs", 0, NULL)` mounts.  If it returns `-EFAULT` (corrupt) or `-EINVAL` (fresh, unformatted chip), retry with `"forceformat"` to format-and-mount.

!!! note "forceformat fallback instead of autoformat"
    NuttX LittleFS's `"autoformat"` option only triggers on `-EFAULT`.  But on a fresh chip (all 0xff), LittleFS finds no superblock and returns `-EINVAL` (`LFS_ERR_INVAL`, see `nuttx/fs/littlefs/littlefs/lfs.c:4259`), so autoformat alone never fires.  This driver tries the initial mount with `NULL` data; on `-EFAULT` or `-EINVAL` it retries with `"forceformat"`.  SPI errors (`-EIO`) do not trigger format (preventing accidental data loss).

### Failure Behavior (non-fatal)

If `stm32_w25q256_initialize()` fails (JEDEC mismatch, SPI error, mount failure, etc.) it logs `syslog(LOG_ERR, ...)` and `stm32_bringup()` continues.  NSH starts with `/mnt/flash` absent.  Check `dmesg` for `W25Q256:` lines to diagnose.

## NVIC IRQ Priority

`stm32_bringup.c` (in the "Issue #36 epsilon plan" block) sets `STM32_IRQ_SPI2` / `STM32_IRQ_DMA1S3` / `STM32_IRQ_DMA1S4` to **NVIC 0xB0**, co-resident with USB OTG FS.  This follows the ε scheme that compresses pybricks's relative ordering (Sound 4 > Flash 5/6 > USB 6 > ADC/TLC 7) into the 0x80-0xF0 BASEPRI band.  See [DMA / IRQ Allocation](../hardware/dma-irq.md) for details.

## Configuration

usbnsh defconfig:

```
# SPI2 + DMA
CONFIG_STM32_SPI2=y
CONFIG_STM32_SPI2_DMA=y

# MTD
CONFIG_MTD=y
CONFIG_MTD_BYTE_WRITE=y
CONFIG_MTD_PARTITION=y

# LittleFS
CONFIG_FS_LITTLEFS=y

# md5 (for tests / verification)
CONFIG_NETUTILS_CODECS=y
CONFIG_CODECS_HASH_MD5=y
```

## Known Limitations

- **DMA priority pinned at MEDIUM** — different from pybricks's HIGH.  See the note in [DMA / IRQ Allocation](../hardware/dma-irq.md).
- **SPI2 bus is single-device** — to add another SPI2 device, extend the `devid` switch in `stm32_spi2select`.
- **Write-throughput limits** — LittleFS does multiple erase + write cycles per metadata update.  First-time file creation can take several seconds (measured: ~1 MB in 45 s ≈ 23 KB/s including Zmodem overhead).
